# 自动更新与发布方案

本文记录 PortBeacon 的两套 GitHub Release 打包方案。

- **方案一**：旧 CI/CD 流程，只负责构建安装包并发布到 GitHub Release，不考虑应用内检查更新。
- **方案二**：当前推荐流程，使用 `tauri-apps/tauri-action` 构建并发布，同时生成应用内检查更新需要的 `latest.json` 和 `.sig`。

## 1. 背景

PortBeacon 是 Tauri 桌面应用，发版可以分成两种目标：

- **普通下载**：用户从 GitHub Release 下载 `.dmg`、`.msi`、`.exe`、`.deb`、`.rpm`、`.AppImage` 等安装包。方案一已经满足这个目标。
- **应用内检查更新**：应用请求 `latest.json`，再根据当前平台下载对应更新包，并使用 `pubkey` 校验 `.sig` 签名。方案二覆盖这个目标。

因此，如果选择方案二支持应用内更新，不是“只上传安装包”就够了，还必须有：

- Tauri updater 公钥：写入 `src-tauri/tauri.conf.json`。
- Tauri updater 私钥：放到 GitHub Actions Secret，不提交到仓库。
- updater artifacts：由 Tauri build 生成更新包和 `.sig`。
- `latest.json`：作为更新索引上传到 GitHub Release。

## 2. 核心原理

本节只说明方案二的应用内检查更新原理。方案一只负责发布普通安装包，不涉及这里的 updater 流程。

应用内手动检查更新的流程如下：

```text
设置页点击“检查更新”
  -> @tauri-apps/plugin-updater check()
  -> 请求 tauri.conf.json 中配置的 latest.json
  -> 解析 version / notes / pub_date / platforms
  -> 当前版本小于远端版本时返回 update
  -> 用户点击“下载并安装”
  -> 下载当前平台对应更新包
  -> 用 tauri.conf.json 中的 pubkey 校验签名
  -> 校验通过后安装
  -> 提示用户重启应用
```

当前 endpoint：

```text
https://github.com/hyq965672903/portbeacon/releases/latest/download/latest.json
```

`GITHUB_TOKEN` 不需要手动配置，它是 GitHub Actions 自动提供的 token。方案二需要额外配置 updater 私钥，具体步骤见 `4.1 签名密钥配置`。

## 3. 方案一：手写 build + release

这是旧方案，也是 `docs/cicd.md` 里最初描述的 CI/CD 流程。它的目标是：**构建安装包并发布到 GitHub Release**。

它不处理应用内检查更新，不生成 `latest.json`，也不关心 updater 签名。

### 3.1 执行流程

```text
推送 v* tag 或手动触发 workflow
  -> build job 按平台 matrix 构建安装包
  -> 上传每个平台的构建产物为 workflow artifact
  -> release job 等 build 完成
  -> 下载所有 workflow artifact
  -> 拼接 .github/release-body.md 和 GitHub 自动变更记录
  -> gh release create 创建 GitHub Release
  -> 上传 .dmg / .zip / .msi / .exe / .deb / .AppImage / .rpm
```

### 3.2 完整文件示例

```yaml
# Workflow 名称，展示在 GitHub Actions 页面。
name: Build Release Bundles

# 触发条件：
# 1. workflow_dispatch 允许在 GitHub 页面手动运行。
# 2. push.tags 只在推送 v 开头的 tag 时自动运行，例如 v0.2.1。
on:
  workflow_dispatch:
  push:
    tags:
      - "v*"

# 允许 workflow 创建 Release、上传 Release assets。
permissions:
  contents: write

# 旧方案拆成两个 job：
# 1. build：各平台打包。
# 2. release：统一创建 GitHub Release。
jobs:
  build:
    name: ${{ matrix.name }}
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        include:
          - name: macOS Apple Silicon
            os: macos-14
            rust-target: aarch64-apple-darwin
            bundle-args: --bundles dmg
            app-zip-name: ""
          - name: macOS Intel
            os: macos-15-intel
            rust-target: x86_64-apple-darwin
            bundle-args: --bundles app
            app-zip-name: portbeacon_0.1.0_x64.app.zip
          - name: Windows x64
            os: windows-latest
            rust-target: x86_64-pc-windows-msvc
            bundle-args: ""
            app-zip-name: ""
          - name: Linux x64
            os: ubuntu-22.04
            rust-target: x86_64-unknown-linux-gnu
            bundle-args: ""
            app-zip-name: ""

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 10

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Setup Rust
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.rust-target }}

      - name: Install Linux dependencies
        if: runner.os == 'Linux'
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            libwebkit2gtk-4.1-dev \
            libgtk-3-dev \
            libayatana-appindicator3-dev \
            librsvg2-dev \
            patchelf

      - name: Install frontend dependencies
        run: pnpm install --frozen-lockfile

      # 使用 Tauri CLI 构建当前 matrix 平台的安装包。
      - name: Build Tauri bundle
        run: pnpm tauri build --target ${{ matrix.rust-target }} --ci ${{ matrix.bundle-args }}

      # 旧方案里额外手写 macOS Intel 的 .app.zip 资产。
      # 这是普通下载资产，不是 updater 必需文件。
      - name: Create macOS app zip
        if: matrix.app-zip-name != ''
        run: |
          cd "src-tauri/target/${{ matrix.rust-target }}/release/bundle/macos"
          ditto -c -k --sequesterRsrc --keepParent portbeacon.app "${{ matrix.app-zip-name }}"

      # 先把各平台产物作为 workflow artifact 上传，给 release job 使用。
      - name: Upload bundles
        uses: actions/upload-artifact@v4
        with:
          name: portbeacon-${{ matrix.rust-target }}
          path: |
            src-tauri/target/${{ matrix.rust-target }}/release/bundle/**/*
            src-tauri/target/release/bundle/**/*
          if-no-files-found: error

  # release job 只负责创建 GitHub Release 和上传普通安装包。
  # 注意：旧方案没有 latest.json / .sig 自动更新链路。
  release:
    name: Publish GitHub Release
    needs: build
    if: startsWith(github.ref, 'refs/tags/v')
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Download bundles
        uses: actions/download-artifact@v4
        with:
          path: release-artifacts

      - name: Create release notes
        env:
          GH_TOKEN: ${{ github.token }}
          TAG_NAME: ${{ github.ref_name }}
          REPOSITORY: ${{ github.repository }}
        run: |
          cp .github/release-body.md release-notes.md
          printf "\n\n## 自动生成的变更记录\n\n" >> release-notes.md
          gh api "repos/${REPOSITORY}/releases/generate-notes" \
            -f tag_name="${TAG_NAME}" \
            --jq ".body" >> release-notes.md

      - name: Publish release
        env:
          GH_TOKEN: ${{ github.token }}
          TAG_NAME: ${{ github.ref_name }}
        run: |
          mapfile -d '' assets < <(
            find release-artifacts -type f \( \
              -name "*.dmg" -o \
              -name "*.zip" -o \
              -name "*.msi" -o \
              -name "*.exe" -o \
              -name "*.deb" -o \
              -name "*.AppImage" -o \
              -name "*.rpm" \
            \) -print0
          )

          if [ "${#assets[@]}" -eq 0 ]; then
            echo "No release assets found."
            exit 1
          fi

          gh release create "${TAG_NAME}" "${assets[@]}" \
            --title "PortBeacon ${TAG_NAME}" \
            --notes-file release-notes.md \
            --verify-tag
```

### 3.3 优缺点

优点：

- 构建和发布拆成两个 job，结构清楚。
- 可以手写特殊产物，例如 `portbeacon_0.1.0_x64.app.zip`。
- 可以完全控制 Release assets 的筛选逻辑。

缺点：

- 不支持应用内检查更新。
- 不会生成 updater 所需的 `latest.json`。
- 不处理 updater `.sig` 签名文件。
- 如果未来要支持自动更新，需要额外写脚本生成并上传 `latest.json`。

## 4. 方案二：tauri-action 发布，当前推荐

当前采用此方案。核心是使用 `tauri-apps/tauri-action` 同时完成构建、上传 release assets、上传 updater signatures 和生成 `latest.json`。

它比方案一更适合 PortBeacon 现在的目标：**GitHub 仍然负责打包发布，同时 App 设置页可以手动检查更新**。

### 4.1 签名密钥配置

方案二需要 updater 签名密钥。方案一只发布普通安装包，不需要这一节。

本地生成 updater key：

```bash
pnpm tauri signer generate -w ~/.tauri/portbeacon.key
```

生成后会得到：

```text
~/.tauri/portbeacon.key      # 私钥，不能提交
~/.tauri/portbeacon.key.pub  # 公钥，可以提交到 tauri.conf.json
```

复制私钥到剪贴板：

```bash
pbcopy < ~/.tauri/portbeacon.key
```

然后在 GitHub 仓库配置 Secret：

```text
Settings
-> Secrets and variables
-> Actions
-> New repository secret
```

新增：

```text
Name: TAURI_SIGNING_PRIVATE_KEY
Value: 粘贴 ~/.tauri/portbeacon.key 的内容
```

公钥写在 `src-tauri/tauri.conf.json`：

```json
{
  "plugins": {
    "updater": {
      "endpoints": [
        "https://github.com/hyq965672903/portbeacon/releases/latest/download/latest.json"
      ],
      "pubkey": "这里是 ~/.tauri/portbeacon.key.pub 的内容"
    }
  }
}
```

> 注意：私钥丢失后，老版本应用无法验证后续更新包。需要妥善备份私钥。

### 4.2 执行流程

```text
推送 v* tag 或手动触发 workflow
  -> release job 按平台 matrix 构建
  -> tauri-action 调用 Tauri build
  -> Tauri 因 createUpdaterArtifacts=true 生成 updater 产物和 .sig
  -> tauri-action 上传普通安装包
  -> tauri-action 上传 .sig
  -> tauri-action 生成并上传 latest.json
  -> 旧版本 App 请求 latest.json 后即可发现新版本
```

### 4.3 完整文件

当前 `.github/workflows/build-release.yml`：

```yaml
# Workflow 名称，展示在 GitHub Actions 页面。
name: Build Release Bundles

# 触发条件：
# 1. workflow_dispatch 允许手动运行。
# 2. push.tags 只在推送 v 开头的 tag 时自动运行。
on:
  workflow_dispatch:
  push:
    tags:
      - "v*"

# tauri-action 需要写权限来创建/更新 Release、上传 assets 和 latest.json。
permissions:
  contents: write

# 当前方案只有一个 release job。
# job id 叫 release，但 matrix 里的每个任务实际都会执行“构建 + 上传”。
jobs:
  release:
    name: ${{ matrix.name }}
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        include:
          - name: macOS Apple Silicon
            os: macos-14
            rust-target: aarch64-apple-darwin
            args: --target aarch64-apple-darwin --bundles dmg
          - name: macOS Intel
            os: macos-15-intel
            rust-target: x86_64-apple-darwin
            args: --target x86_64-apple-darwin --bundles app
          - name: Windows x64
            os: windows-latest
            rust-target: x86_64-pc-windows-msvc
            args: --target x86_64-pc-windows-msvc
          - name: Linux x64
            os: ubuntu-22.04
            rust-target: x86_64-unknown-linux-gnu
            args: --target x86_64-unknown-linux-gnu

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 10

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Setup Rust
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.rust-target }}

      - name: Install Linux dependencies
        if: runner.os == 'Linux'
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            libwebkit2gtk-4.1-dev \
            libgtk-3-dev \
            libayatana-appindicator3-dev \
            librsvg2-dev \
            patchelf

      - name: Install frontend dependencies
        run: pnpm install --frozen-lockfile

      # 读取手写 release body，作为 GitHub Release 说明的前置内容。
      - name: Read release body
        id: release_body
        shell: bash
        run: |
          {
            echo 'body<<EOF'
            cat .github/release-body.md
            echo
            echo 'EOF'
          } >> "$GITHUB_OUTPUT"

      # tauri-action 负责：
      # 1. 调用 Tauri build。
      # 2. 上传普通安装包。
      # 3. 上传 updater .sig。
      # 4. 生成并上传 latest.json。
      - name: Build and publish release
        uses: tauri-apps/tauri-action@v0
        env:
          # GitHub 自动提供，不需要手动配置。
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          # 需要在 GitHub Actions Secrets 中手动配置。
          # 这是 ~/.tauri/portbeacon.key 的内容，不能提交到仓库。
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
        with:
          tagName: ${{ github.ref_name }}
          releaseName: PortBeacon ${{ github.ref_name }}
          releaseBody: ${{ steps.release_body.outputs.body }}
          generateReleaseNotes: true
          releaseDraft: false
          prerelease: false
          args: ${{ matrix.args }}
          # 生成/上传 latest.json，供 App 检查更新。
          uploadUpdaterJson: true
          # 上传 .sig，供 App 校验更新包签名。
          uploadUpdaterSignatures: true
```

### 4.4 配套 tauri.conf.json

关键配置：

```json
{
  "bundle": {
    "active": true,
    "targets": "all",
    "createUpdaterArtifacts": true
  },
  "plugins": {
    "updater": {
      "endpoints": [
        "https://github.com/hyq965672903/portbeacon/releases/latest/download/latest.json"
      ],
      "pubkey": "这里是 updater 公钥"
    }
  }
}
```

`createUpdaterArtifacts` 的作用是让 Tauri 生成 updater 需要的更新包和 `.sig` 签名文件。

### 4.5 优缺点

优点：

- 自动上传 updater signatures。
- 自动生成并上传 `latest.json`。
- 发版链路更短，少写自定义脚本。
- 更贴近 Tauri 官方推荐路径。

缺点：

- Release asset 命名更多交给 Tauri/tauri-action 决定。
- 如果要保留手写的特殊文件名，例如 `portbeacon_0.1.0_x64.app.zip`，需要额外步骤。
- 多平台 matrix 同时发布同一个 Release 时，需要关注并发上传。如果遇到并发冲突，可以加 `strategy.max-parallel: 1` 让平台串行发布。

当前先保持并行 matrix，让 GitHub 打包速度更快。如果实际发版时发现 Release asset 或 `latest.json` 上传冲突，再优先加：

```yaml
strategy:
  max-parallel: 1
```

## 5. 两种方案对比

| 项目 | 方案一：手写 build + release | 方案二：tauri-action |
| --- | --- | --- |
| 普通安装包 | 支持 | 支持 |
| GitHub Release | 手写 `gh release create` | `tauri-action` 处理 |
| `latest.json` | 不考虑 | 可自动生成 |
| `.sig` 上传 | 不考虑 | 可自动上传 |
| 特殊资产命名 | 最灵活 | 需要额外步骤 |
| 应用内检查更新 | 不支持 | 支持 |
| 维护成本 | 高 | 低 |
| 当前推荐 | 否 | 是 |

## 6. 发版流程

1. 更新版本号：
   - `package.json`
   - `src-tauri/Cargo.toml`
   - `src-tauri/Cargo.lock`
   - `src-tauri/tauri.conf.json`
2. 更新 `.github/release-body.md`。
3. 提交代码。
4. 创建并推送 tag：

```bash
git tag v0.2.1
git push origin master
git push origin v0.2.1
```

5. GitHub Actions 自动打包并发布 Release。
6. Release assets 中应包含普通安装包、`.sig` 和 `latest.json`。
7. 旧版本 App 的设置页点击“检查更新”，应能读取 `latest.json` 并提示新版本。

## 7. 常见问题

### 7.1 `Could not fetch a valid release JSON from the remote`

原因通常是：

- Release 里没有 `latest.json`。
- `latest.json` 不是 Tauri updater 需要的格式。
- endpoint 配错。
- Release 还没发布完成。

处理方式：

- 确认 GitHub Release assets 里有 `latest.json`。
- 确认 `src-tauri/tauri.conf.json` 的 endpoint 指向该文件。
- 确认 workflow 使用 `uploadUpdaterJson: true`。

### 7.2 `GITHUB_TOKEN` 需要手动配置吗？

不需要。`GITHUB_TOKEN` 是 GitHub Actions 自动提供的内置 token。

需要手动配置的是：

```text
TAURI_SIGNING_PRIVATE_KEY
```

### 7.3 没有 `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` 可以吗？

可以。当前私钥生成时没有设置密码，所以 workflow 不配置该变量。

### 7.4 新 Release 还会有普通安装包吗？

会。普通用户下载的 `.dmg`、`.msi`、`.exe`、`.deb`、`.rpm`、`.AppImage` 等安装包仍会作为 Release assets 上传。

自动更新会额外需要：

```text
latest.json
*.sig
```

### 7.5 还会有 `Source code (zip)` 和 `Source code (tar.gz)` 吗？

会。这是 GitHub 对每个 tag 自动生成的源码压缩包，和 workflow 方案无关。

### 7.6 还会有旧 workflow 的 `portbeacon_0.1.0_x64.app.zip` 吗？

不保证。那个文件是旧 workflow 手写 `ditto` 生成的特殊资产。当前方案交给 `tauri-action` 和 Tauri bundler 处理，文件名可能变化。如果必须保留这个资产，需要额外加压缩和上传步骤。

    # 端口归因分析设计

PortBeacon 的端口归因不是单一规则判断，而是一个分层分析管线。目标是在默认「开发端口」视图里尽量展示用户真正关心的开发服务，同时把系统服务、生活软件、工具后台等噪声默认折叠到「全部端口」里。

## 核心目标

1. 不只展示 `node PID 12345`，而是解释“谁拉起了这个端口”。
2. 不只依赖端口号，因为 `8000 / 8080 / 8081 / 8888` 可能来自 Node、Java、Python、Go、Docker 或工具后台。
3. 不把父链里的 IDE / AI 工具误判成负证据。比如 `Codex -> IntelliJ IDEA -> Java` 应归因给 IntelliJ IDEA，Codex 只是上游来源。
4. 默认视图只展示高相关开发端口，折叠项仍然可以在「全部端口」里看到。

## 当前模块结构

```text
src-tauri/src/modules/
  port/
    scanner.rs       # 扫描端口事实
    process.rs       # 读取进程事实、进程链、停止进程
    monitor.rs       # 端口变化监控
    service.rs       # 端口列表业务编排
    model.rs         # 端口相关 QO / VO / 快照模型

  analysis/
    model.rs         # 0. 分析模型
    engine.rs        # 1. 分析入口和流程编排
    source.rs        # 2. 来源追溯
    context.rs       # 3. 上下文补全
    fingerprint.rs   # 4. 服务指纹
    scoring.rs       # 5. 归因打分
    classification.rs # 6. 硬规则和分类决策
    feedback.rs      # 7. 用户修正覆盖入口
    portbeacon-rules.default.json # 默认用户规则模板
    postprocess.rs   # 8. 同一进程多端口集合后处理
```

边界原则：

```text
port = 采集事实
analysis = 解释事实
```

## 执行顺序

```text
端口快照
  -> 0. model：分析模型承载
  -> 1. engine：分析入口和流程编排
  -> 2. source：进程链来源追溯
  -> 3. context：上下文补全（runtime / cwd / project）
  -> 4. fingerprint：服务指纹补证据
  -> 5. scoring：分组证据打分（owner / source / project / service / internal）
  -> 6. classification：硬规则和分类决策（内部先硬规则，再使用评分和指纹结果）
  -> 7. feedback：用户修正规则覆盖
  -> 8. postprocess：同一进程多端口集合后处理
  -> UI 输出
```

当前代码里的实际入口是：

```rust
analysis::engine::infer_port_context(...)
```

`port/process.rs` 负责收集进程事实，然后把证据交给 analysis：

```text
build_service
  -> collect_process_chain
  -> ProcessEvidence
  -> infer_port_context
```

## 步骤 2：来源追溯

来源追溯回答的是“谁拉起来的”，不等于“是否开发端口”。

典型链路：

```text
Codex -> IntelliJ IDEA -> Java
```

归因应该是：

```text
owner: Java
direct source: IntelliJ IDEA
upstream source: Codex
```

摘要应表达为：

```text
IntelliJ IDEA 启动了 Java，占用了 :8888。上游进程包含 Codex。
```

而不是：

```text
Codex 启动了 Java。
```

注意：父链里的 IDE / AI 工具不能直接当负证据。

## 步骤 3：上下文补全

上下文补全负责从进程事实里识别：

```text
runtime
launcher
framework
project
```

项目上下文不依赖 `/work/ /project/ /dev/` 这类本机目录名，而是从 `cwd` 或命令行路径向上查找项目标记文件，例如：

```text
package.json
pom.xml
go.mod
Cargo.toml
pyproject.toml
```

## 步骤 4：服务指纹

服务指纹用于补证据。

当前实现包含两类安全指纹：

```text
Docker / OrbStack + 3306  => MySQL
Docker / OrbStack + 5432  => PostgreSQL
Docker / OrbStack + 6379  => Redis
Docker / OrbStack + 27017 => MongoDB

AI Agent 来源端口
=> 对 127.0.0.1:<port> 做 120ms 短超时 HTTP HEAD 探测
=> 能返回 HTTP 响应时补充 HTTP service 证据
```

注意：HTTP 指纹不是端口号白名单，不写死 `18789` 这类现场端口。某个 AI Agent 端口是否进入开发端口，取决于它是否能提供通用服务证据；不能响应 HTTP 的辅助端口仍会默认折叠。

主动指纹可以在设置页关闭。后续如果继续扩展更深的 banner 探测，需要继续遵守：

```text
必须限制 localhost / 127.0.0.1 / ::1，设置短超时，并允许用户关闭
```

## 步骤 5：归因打分

模糊场景用分组证据打分，不用单个 if/else 一刀切。

当前分组：

```text
owner     # 监听进程本身是否像服务进程
source    # 来源是否像 IDE / AI 工具 / Terminal
project   # 是否有项目目录或项目文件上下文
service   # 是否命中 launcher / framework / known service / 常见服务端口画像
internal  # 是否有系统目录、应用包内部等负向证据
```

例子：

```text
owner 是 Java / Node.js / Python
=> owner 加分

来源是 IntelliJ IDEA / WebStorm / Codex / Antigravity / Terminal
=> source 加分

cwd 或命令行路径向上能找到 package.json / pom.xml / go.mod 等项目标记文件
=> project 加分

命中 pnpm dev / Gradle / Uvicorn / Vite / Spring Boot
=> service 加分

命中 3000 / 5173 / 8080 / 8888 等常见开发服务端口画像
=> service 小幅加分

owner 位于 /usr/libexec 或 /System/Library
=> internal 减分
```

这层的目的不是替代硬规则，而是处理硬规则无法覆盖的开发现场。

开发端口成立条件不能再使用 `owner 高 + source 高 => development`。更稳的条件是：

```text
owner 像服务进程
并且具备 service 证据
并且具备 project 证据，或者具备强 service 证据
```

注意：端口号不能单独作为强裁决条件。比如 49152-65535 动态/私有端口范围只能作为辅助信号，不能因为端口在这个范围里就直接排除。真正要判断这类端口是不是内部辅助端口，需要结合“同一 PID 是否还有更明确的主服务端口”一起看。

## 步骤 6：硬规则和分类决策

分类模块内部先处理高确定性硬规则，再使用 scoring 和 fingerprint 的结果做模糊分类。

适合硬规则的情况：

```text
owner 是系统进程
=> collapsed / system-service

owner 是生活软件或浏览器本体
=> collapsed / lifestyle-app 或 browser

owner 自身位于 /Applications/*.app/Contents，且 owner 就是 IDE / AI IDE / 工具本体
=> collapsed / tool-background
```

## 步骤 7：用户修正学习

用户修正用于吃掉系统猜错的少数场景。

当前入口：

```text
系统设置 -> 归因修正规则
新增端口规则
启用 / 禁用规则
删除规则
```

存储：

```text
portbeacon-rules.json
```

当前文件会在 Tauri app data 目录下初始化。示例：

开发期默认模板在仓库中：

```text
src-tauri/src/modules/analysis/portbeacon-rules.default.json
```

首次启动时会复制到 Tauri app data 目录：

```text
~/Library/Application Support/<identifier>/portbeacon-rules.json
```

之后设置页编辑和外部修改都作用于 app data 下的运行时副本，不直接改仓库模板。示例：

```json
[
  {
    "id": "collapse-rustrover-helper",
    "name": "折叠 RustRover 工具后台",
    "enabled": true,
    "matcher": {
      "executableIncludes": ["RustRover.app"]
    },
    "category": "tool-background",
    "visibility": "collapsed",
    "reason": "用户标记为工具后台"
  },
  {
    "id": "focus-backend-8888",
    "name": "后端 8888 开发服务",
    "enabled": true,
    "matcher": {
      "ports": [8888],
      "processNameIncludes": ["java"]
    },
    "category": "dev-server",
    "visibility": "focused",
    "reason": "用户标记为开发端口"
  }
]
```

生成规则要保守。比如用户选择“默认折叠同类”时，优先生成：

```text
executable contains RustRover.app
```

不要生成：

```text
processName = rust
```

否则可能误伤真正的 Rust 项目服务。

## 步骤 8：同一进程多端口集合后处理

单个端口分析完成后，列表会在筛选前做一次集合后处理，用同一 PID 的多端口关系修正默认展示结果。

典型场景：

```text
PID 46992 / Java
  :8888   业务端口
  :53493  JVM / IDE 辅助端口
```

单端口分析只能看到两者都是 `Java + IDEA`，容易把两个都判成开发端口。集合后处理会保留更像主服务的端口，并把缺少明确服务证据的动态辅助端口默认折叠。

## visibility 语义

当前 visibility 有两个值：

```text
focused
collapsed
```

含义：

```text
focused
=> 进入默认「开发端口」视图

collapsed
=> 默认开发端口视图里折叠
=> 仍然会出现在「全部端口」里
=> 搜索也应该能搜到
```

不要把 `collapsed` 理解成“扫描不到”或“彻底隐藏”。

## 设计原则

```text
硬规则处理确定性
打分处理模糊性
指纹补充服务证据
来源追溯负责解释链路
用户反馈负责长期纠错
```

一句话：

```text
规则驱动 + 证据打分 + 归因追溯
```
